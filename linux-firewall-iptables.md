## Netfilter — фундамент всего

**Netfilter** — это подсистема внутри ядра Linux, появившаяся в версии 2.4. Она регистрирует **хуки** (hook points) в сетевом стеке ядра: PREROUTING, INPUT, FORWARD, OUTPUT, POSTROUTING. Через эти хуки проходит **каждый пакет** — Netfilter буквально вшит в сетевой стек ядра. Сам по себе он ничего не блокирует — он лишь предоставляет инфраструктуру для других инструментов.

Важно понять: **Netfilter — это ядро**, а `iptables`, `nftables`, `firewalld` — лишь пользовательские интерфейсы к нему. Это как ядро базы данных и клиент к ней — они не одно и то же.

---

## Два бэкенда внутри Netfilter

Это ключевой момент, который часто путают:

|  | **iptables-backend** | **nf_tables backend** |
| --- | --- | --- |
| Появился | Ядро 2.4 (2001) | Ядро 3.13 (2014) |
| Используется | iptables-legacy | nftables / nft |
| Принцип | Линейный список правил | Байткод (как JIT) |
| Оба активны одновременно? | ✅ Да, технически возможно | — |

Оба бэкенда могут работать одновременно, и тогда пакет должен пройти через **оба фаервола**. Именно поэтому смешивать iptables и nftables на одном сервере — плохая практика.

## Инструменты пользовательского пространства

### iptables

Классический инструмент, появился вместе с Netfilter. Пишешь правило командой `iptables -A INPUT ...` — оно транслируется в ядро через старый бэкенд. Проблемы: отдельные утилиты для IPv4 (`iptables`), IPv6 (`ip6tables`), ARP (`arptables`), Ethernet-кадров (`ebtables`). Четыре инструмента вместо одного — сложно и громоздко.

### nftables / nft

`nftables` — это **новый бэкенд** в ядре, а `nft` — утилита для работы с ним. Один инструмент заменяет все четыре утилиты iptables. Синтаксис ближе к языку программирования, поддерживаются переменные, sets, maps. Правила компилируются в байткод — работает быстрее на большом количестве правил.

```bash
# iptables — старый синтаксис
iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# nft — новый синтаксис
nft add rule inet filter input tcp dport 22 accept
```

### firewalld

**Высокоуровневый менеджер** с поддержкой зон (`public`, `internal`, `dmz`). Под капотом использует nftables (начиная с RHEL 8) или iptables. Удобен для управления сложными инфраструктурами — не надо думать о низкоуровневых цепочках. Стандарт на RHEL/CentOS/Fedora.

### ufw (Uncomplicated Firewall)

Ещё один высокоуровневый frontend, ориентированный на простоту. Стандарт на Ubuntu. Под капотом — iptables или nftables.

## Практический вывод для DevOps

- На **новых серверах** (Debian 12+, Ubuntu 22.04+, RHEL 9) — учи `nftables` / `nft`, это настоящее и будущее
- На **legacy-серверах** — iptables ещё жив, знать его обязательно
- **Никогда не смешивай** iptables и nftables на одном сервере — будет два независимых фаервола
- `firewalld` / `ufw` — удобны для старта, но в продакшене DevOps часто работает с `nft` или `iptables` напрямую для полного контроля

## Жизненный цикл пакета (5 хуков)

Пакет в Linux проходит через фиксированный конвейер хуков — **один раз и строго по порядку**. Если правило стоит не в том месте — оно не сработает никогда, даже если синтаксически корректно.

```
Входящий пакет с сети
        │
   [PREROUTING]   ← пакет только пришёл, маршрут ещё не выбран
        │
   Решение о маршруте
       / \
      /   \
 [INPUT]  [FORWARD]   ← INPUT = для нас, FORWARD = транзит
    │          │
Локальный   [POSTROUTING] → уходит дальше
 процесс
    │
 [OUTPUT]    ← наш процесс отправляет пакет
    │
[POSTROUTING] → уходит в сеть
```

## Таблицы iptables

Таблицы определяют **цель обработки** пакета:

| Таблица | Назначение | Пример использования |
| --- | --- | --- |
| `filter` | Фильтрация — разрешить или заблокировать | Закрыть порт 22 для всех кроме своего IP |
| `nat` | Трансляция адресов | Проброс порта, маскарадинг (Docker, роутер) |
| `mangle` | Модификация пакетов | Изменить TTL, метки QoS |
| `raw` | Работа с сырыми пакетами до conntrack | Обход отслеживания соединений |

---

## Цепочки таблицы filter

Самая используемая таблица — `filter`, она содержит три цепочки:

- **INPUT** — пакеты, адресованные **этому серверу** (входящий трафик к вашим сервисам)
- **OUTPUT** — пакеты, **исходящие** от этого сервера
- **FORWARD** — пакеты, которые **проходят транзитом** (сервер как роутер)

## Первые команды

```bash
# Посмотреть текущие правила таблицы filter
iptables -L -v -n

# Посмотреть правила в виде команд (удобно для копирования)
iptables -S

# Добавить правило: разрешить входящий TCP на порт 22
iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# Заблокировать входящий трафик с конкретного IP
iptables -A INPUT -s 1.2.3.4 -j DROP

# Удалить всё (сброс правил)
iptables -F
```

Ключ `-A` (append) добавляет правило в **конец** цепочки, а `-I` (insert) — в **начало**. Порядок правил критичен: пакет проверяется сверху вниз и останавливается на первом совпадении.

## Ключевой принцип: Stateful vs Stateless

Модуль `conntrack` (connection tracking) позволяет iptables отслеживать **состояние соединений**:

- `NEW` — новое соединение
- `ESTABLISHED` — уже установленное соединение
- `RELATED` — связанное с существующим (например, FTP data-channel)
- `INVALID` — некорректный пакет

Базовое stateful-правило, которое стоит в любом продакшн-сервере:

```bash
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
```

## Практика: защита сервера с нуля

### Важно

**Никогда не ставь `DROP` по умолчанию первой командой** — если ты работаешь по SSH, то сразу себя заблокируешь. Правильный порядок: сначала пишем все разрешающие правила, потом ставим `DROP`.

### Шаг 1 — Сброс и подготовка

```bash
# Сначала ставим ACCEPT — чтобы не потерять SSH во время настройки
iptables -P INPUT ACCEPT
iptables -P OUTPUT ACCEPT
iptables -P FORWARD ACCEPT

# Очищаем все существующие правила и цепочки
iptables -F   # flush — удалить все правила
iptables -X   # delete — удалить пользовательские цепочки
```

### Шаг 2 — Базовый набор правил

Пишем правила в правильном порядке: **от важных к специфичным**:

```bash
# 1. Разрешить loopback (интерфейс lo — 127.0.0.1)
# Нужен для работы ЛЮБЫХ локальных сервисов: БД, Docker, systemd
iptables -A INPUT -i lo -j ACCEPT

# 2. Сбросить INVALID-пакеты (не принадлежат ни одному известному соединению)
# Защита от TCP-спуфинга, сломанных пакетов
iptables -A INPUT -m conntrack --ctstate INVALID -j DROP

# 3. Принять уже УСТАНОВЛЕННЫЕ и СВЯЗАННЫЕ соединения
# Это ключевое правило — без него ответы на твои запросы будут блокироваться!
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT

# 4. Разрешить SSH (порт 22 — новые подключения)
iptables -A INPUT -p tcp --dport 22 -m conntrack --ctstate NEW -j ACCEPT

# 5. Разрешить HTTP/HTTPS (если это веб-сервер)
iptables -A INPUT -p tcp --dport 80 -m conntrack --ctstate NEW -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -m conntrack --ctstate NEW -j ACCEPT

# 6. Разрешить ping с ограничением — не более 1 запроса в секунду
iptables -A INPUT -p icmp --icmp-type echo-request -m limit --limit 1/sec -j ACCEPT

# 7. ТОЛЬКО ТЕПЕРЬ — установить DROP по умолчанию
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT  # исходящий трафик обычно не ограничивают
```

### Шаг 3 — Проверить результат

```bash
# Детальный вывод: счётчики пакетов, байт, интерфейсы
iptables -L -v -n

# Вывод в виде команд — легко скопировать и воспроизвести
iptables -S
```

Ожидаемый вывод `iptables -S`:

```
-P INPUT DROP
-P FORWARD DROP
-P OUTPUT ACCEPT
-A INPUT -i lo -j ACCEPT
-A INPUT -m conntrack --ctstate INVALID -j DROP
-A INPUT -m conntrack --ctstate RELATED,ESTABLISHED -j ACCEPT
-A INPUT -p tcp --dport 22 -m conntrack --ctstate NEW -j ACCEPT
...
```

### Шаг 4 — Сохранить правила

По умолчанию правила iptables **исчезают после перезагрузки**. Чтобы сохранить:

```bash
# Debian/Ubuntu
apt install iptables-persistent
netfilter-persistent save

# Или вручную
iptables-save > /etc/iptables/rules.v4

# Восстановить вручную
iptables-restore < /etc/iptables/rules.v4
```

## Визуально: что происходит с пакетом

```
Входящий пакет
      │
      ▼
[lo интерфейс?] ──YES──► ACCEPT
      │NO
      ▼
[INVALID?] ──YES──► DROP
      │NO
      ▼
[ESTABLISHED/RELATED?] ──YES──► ACCEPT
      │NO
      ▼
[NEW + порт 22/80/443?] ──YES──► ACCEPT
      │NO
      ▼
[политика по умолчанию] ──────► DROP
```

---

## Типичная ошибка

Правило `ESTABLISHED,RELATED` должно идти **до** `DROP`-политики, но **после** блокировки `INVALID`. Если поставить `ESTABLISHED` первым — `INVALID`-пакеты могут проскочить, если последним — обратный трафик заблокируется.
